# Claude Code向け追加指示

## 基本方針

プロジェクトの目的、スコープ、責務境界、安全条件はルートの`AGENTS.md`に従ってください。

このファイルにはClaude Codeで作業する際の追加手順だけを記載します。`AGENTS.md`と同じ内容を重複して管理しません。

## 作業開始時

変更前に次を確認してください。

1. `AGENTS.md`
2. `README.md`
3. `package.json`
4. `tsconfig.json`
5. 対象機能に関連する仕様書とADR
6. 対象コードと既存テスト
7. `git status -sb`

リポジトリに存在しない構成、ライブラリ、コマンドを推測しないでください。文書と実際の構成が異なる場合は、実在するコードを確認し、矛盾を報告してください。

## 実装前の整理

変更を始める前に、必要な範囲で次を整理してください。

- 解決する問題
- 変更対象
- 現在の実装
- 変更方針
- 影響する責務境界
- 実行する確認

小さな修正に長い計画書は不要です。権限、データ保持、通信、ZIP形式、自動操作へ影響する場合は、先に関連ADRの必要性を判断してください。

## 変更範囲

ユーザーが依頼した範囲を超えて変更しないでください。

特に次を避けます。

- 無関係なリファクタリングや命名変更
- ファイル全体の不要な整形
- 将来用の抽象化や未使用コード
- 必要性を確認していない依存関係
- Discord以外への汎用化
- ユーザーの未コミット変更の上書き

別の問題を見つけても、現在のタスクに直接関係しなければ報告に留めます。

## コード編集

既存パターンと責務境界を確認してから編集してください。

- `domain`へDOM、Chrome API、WXTの具体実装を持ち込まない
- Discord DOM変更はextractorと匿名化fixtureを中心に扱う
- Chrome APIの失敗とservice worker再起動を考慮する
- 大容量ZIP処理ではメモリ一括化を避ける
- メッセージ、URL、ファイル名など外部入力を受信側で再検証する
- UIへ動的値を挿入する場合は安全なDOM APIを使う

既存コードを確認せずに、大量のファイルを一括生成しないでください。

## 依存関係

新しい依存を追加する前に次を確認してください。

1. Web API、Chrome API、Node.js標準API、既存依存で実現できないか
2. runtime dependencyとして必要か
3. Manifest V3とContent Security Policyに適合するか
4. リモートコードや動的評価を持ち込まないか
5. ライセンス、lockfile、配布物への影響を確認できるか

依存関係を追加した場合は、理由と配布物への影響を完了報告へ記載してください。

## コマンド実行

利用可能なコマンドは`package.json`から確認します。存在しないコマンドを推測して実行しないでください。

まず変更に近いテストを実行し、問題がなければ変更規模に応じて全体確認を行います。

コマンドが失敗した場合は、次を隠さず報告してください。

- 実行したコマンド
- 失敗内容
- 今回の変更との関係
- 未解決事項

文書だけの変更では、少なくともPrettierと`git diff --check`を確認します。

## テストと実機確認

テストは内部実装ではなく、利用者から見た振る舞いを確認してください。

Discord本番への通信や認証情報を必要とするテストを通常のtest suiteへ追加しません。Happy DOMで再現できないDiscordのvirtualized list、CSS、Chrome downloads、OPFS、権限UIは、自動検証の限界を明記します。

実機確認が必要でも、ユーザーから実施報告を受けていない項目を「確認済み」と記録しないでください。

## Git操作

ユーザーから明示的に依頼されていない限り、次を行いません。

- commit
- push
- branch作成や切り替え
- Pull Request作成
- rebase、reset、既存commitの書き換え
- 未コミット変更の破棄

公開を依頼された場合も、作業前後に差分とブランチを確認し、今回の変更だけをstageします。

## 危険な操作

明示的な必要性と許可がない限り、次を行わないでください。

- `rm -rf`
- `git reset --hard`
- `git clean`
- OPFSやダウンロード済みファイルの一括削除
- 既存設定や文書の全面置換
- 大量ファイルの移動

破壊的操作が必要な場合は、対象と影響を説明してから進めます。

## セキュリティ

トークン、Cookie、Authorization Header、署名付きURL、個人情報をコード、fixture、ログ、文書、完了報告へ含めないでください。

実DOMをユーザーから受け取った場合は、必要な構造だけをテストへ再現し、実在IDやURLを転記しません。

## ドキュメント

利用者向けREADMEは簡潔に保ち、詳細設計、検証記録、ADR、開発手順は`docs/`へ置きます。

内部実装だけの変更でREADMEを更新する必要はありません。利用方法、権限、データ保持、配布方法が変わる場合は、対応する文書を更新してください。

## 完了報告

作業完了時は簡潔に次を報告してください。

- 変更内容
- 主な変更ファイル
- 実行した確認と結果
- 未確認事項や残課題
- commitやPRの状態

確認できなかった項目を推測で「問題なし」としないでください。

## 指示が競合した場合

次の順序で扱います。

1. ユーザーが今回明示した依頼
2. 対象ディレクトリにある、より範囲の狭い指示
3. この`CLAUDE.md`
4. ルートの`AGENTS.md`
5. READMEとその他の文書

秘密情報の保護、ユーザーデータの保全、破壊的操作の回避は常に優先してください。
