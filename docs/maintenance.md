# 保守・更新方針

運用形態は[ADR-0002](adr/0002-continue-limited-unpacked-distribution.md)に基づき、少人数への unpacked 配布を継続します。

## 対象

Chrome Web Storeへ公開せず、信頼できる少人数へunpacked extensionとして限定配布します。自動更新は提供しません。

## 責任者

- Decision owner: Project owner
- 配布、更新通知、問い合わせ一次対応: Project owner
- 配布対象者の追加: Project ownerの明示承認が必要

具体的な担当者名と連絡手段は、公開リポジトリへ個人情報を記載せず、配布時の連絡経路で共有します。

## サポートするバージョン

- 原則として最新の限定配布版だけをサポートする
- セキュリティ修正時は旧版の利用停止を通知する
- 更新できない利用者は、問題が解消するまで拡張機能を無効化する

## 更新フロー

1. `develop`から作業ブランチを作る。
2. 実装、テスト、ドキュメントを更新する。
3. `pnpm release:prepare`で検証、ZIP、SHA-256を生成する。
4. 権利が明確なテストサーバーで手動スモークテストする。
5. Pull Requestをレビューして`develop`へマージする。
6. 配布版のコミットを`main`へ反映する。
7. ZIP、SHA-256、リリースノートを同じ連絡経路で配布する。
8. 利用者へバージョン、権限差分、更新期限を通知する。

## 定期確認

固定のリリース周期は設けません。各リリース前と、Discord または Chrome の仕様・規約変更を把握した時に次を確認します。

- ADR-0001 の機能境界を維持している
- manifest の権限が増えていない
- 配布対象者全員へ直接連絡できる
- 最新版の ZIP、SHA-256、リリースノートを対応付けて保管している
- 依存関係と開発ツールの更新可否を確認した
- 権利が明確なチャンネルで手動スモークテストできる

更新がなくても、Discord の DOM 変更を利用者から報告された場合は障害として扱います。

## 問い合わせと障害対応

問い合わせでは、メッセージ本文、ユーザー名、チャンネル名、完全な URL、ダウンロード先を収集しません。再現に必要な場合は、架空データによる最小 fixture を作ります。

1. 影響するバージョン、Chrome と OS、URL を含まない操作手順を確認する。
2. 誤取得、情報漏えい、権限逸脱の可能性があれば、全利用者へ直ちに無効化を案内する。
3. 権利が明確なテスト環境で再現し、匿名化 fixture と失敗テストを追加する。
4. 修正後に `pnpm release:prepare` と手動スモークテストを実行する。
5. 新しい ZIP、SHA-256、リリースノートと更新期限を通知する。

正式な SLA は設けません。安全性を判断できない間は、機能を継続利用させず無効化を優先します。

## Discord DOM変更時

- メッセージ表示領域を安全に特定できなければ、候補ゼロではなく解析失敗として停止する
- セレクターの探索範囲をページ全体へ広げて回避しない
- 実在チャンネルのHTMLをそのまま保存・共有しない
- 最小化・匿名化したfixtureと再現テストを先に追加する
- 修正版の配布までは拡張機能の利用停止を案内する

## ロールバック

- 配布したZIP、SHA-256、リリースノートをバージョン単位で保管する
- 問題のある版は配布を停止し、利用者へ無効化または直前版への戻し方を通知する
- 直前版へ戻しても安全でない場合は、修正版まで無効化する
- 秘密鍵、署名付きDiscord URL、テストチャンネルの内容をリリース保管物へ含めない

## バージョンと保管

- バージョンは Semantic Versioning で付ける
- 利用者向け変更を `CHANGELOG.md` とリリースノートへ記録する
- 最新版と、安全に戻せる直前版の配布物を最低限保持する
- 問題が判明した版は「利用停止」と明記し、再配布しない
- 保管物の SHA-256 を再計算し、記録と一致することを確認できる状態にする

## 終了条件

次のいずれかに該当した場合、限定配布を停止します。

- Discordの規約・仕様変更により限定用途を維持できない
- 必要権限が現在の方針より大幅に広がる
- DOM変更へ継続的に追随する担当者を確保できない
- 重大な情報漏えいまたは誤取得を安全に修正できない

停止時は全利用者へ無効化・削除を案内し、配布場所から問題のある成果物を取り下げます。利用者データをサーバー側に保持しないため、プロジェクト側で削除する利用データはありません。
