# Contributing

Discord Media Exporterへ機能追加や修正を行う際の開発ルールです。

AI開発ツールは、ルートの[`AGENTS.md`](../AGENTS.md)を共通ルールとして参照します。Claude Code向けの追加手順は[`CLAUDE.md`](../CLAUDE.md)に分離しています。

## 基本方針

- 一つの変更を小さく保つ
- 既存の実装パターンを優先する
- 必要になるまで抽象化しない
- ユーザーが表示したDiscordメディアだけを扱う
- 権限、通信先、保存データを必要最小限にする

## 開発フロー

1. [プロダクト要件](product-requirements.md)と関連仕様を確認する。
2. 設計判断が必要なら、実装前にADRを作成する。
3. 作業用ブランチで実装する。
4. 変更に対応するテストを追加・更新する。
5. lint、型検査、テスト、ビルドを実行する。
6. README、仕様書、リリースノートを必要に応じて更新する。
7. 変更の目的と検証結果を記載してPull Requestを作成する。

## 開発環境

- Node.js 24
- pnpm 9.7.0
- Google Chrome

```bash
pnpm install --frozen-lockfile
pnpm dev
```

詳しい構成とローカル確認方法は[開発ガイド](development.md)を参照してください。

## コーディング

- TypeScriptのstrict設定を維持する
- `any`の使用を避ける
- 関数とクラスの責務を小さく保つ
- 大きなファイルはストリームとして処理する
- エラーを握りつぶさず、利用者向けメッセージへ変換する
- タスクと無関係なリファクタリングや一括整形を混ぜない

DiscordのDOMは変更される可能性があります。class名だけに依存せず、意味のある属性やDOM構造を優先してください。

## テスト

変更前後で次を実行します。

```bash
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```

リリース候補では、配布ZIPとSHA-256の検証も含めて実行します。

```bash
pnpm release:prepare
```

DOM抽出や操作を変更する場合は、再現用fixtureと単体テストを追加してください。実在するDiscordチャンネルのHTML、ユーザー名、本文、ID、署名付きURLはコミットせず、必要な構造だけを匿名化して再現します。

ChromeやDiscordの実際の動作に依存する変更は、自動テストに加えて権利が明確なテスト用チャンネルで確認します。

## ドキュメントとADR

次の変更では関連ドキュメントを更新します。

- ユーザー操作や画面表示
- ManifestとChrome権限
- Discord DOMの抽出条件
- ダウンロードとZIP形式
- 保存領域、通信先、データ保持
- ビルド、配布、リリース手順

複数の実装方法から選択する変更、セキュリティ境界を変える変更、後から覆すコストが高い変更では[ADR](adr/)を追加します。

## コミットとPull Request

コミットは一つの目的ごとに作成し、変更内容が分かる短いメッセージを付けます。

Pull Requestには次を記載します。

- 目的と背景
- 主な変更
- 利用者への影響
- 不具合修正の場合は原因
- 自動テストと実機確認の結果
- 関連IssueやADR

レビューしやすい範囲を保ち、無関係な変更を含めないでください。

## セキュリティ

次の情報をコード、fixture、ログ、Issue、Pull Requestへ含めてはいけません。

- Discordトークン
- CookieやAuthorization Header
- APIキーや秘密鍵
- 実在ユーザーの個人情報
- 不要な署名付きメディアURL

権限や通信先を追加する場合は、必要性、利用タイミング、解放方法を説明し、[セキュリティとプライバシー](security-and-privacy.md)を更新します。

## リリース前チェック

- [ ] lint、型検査、テスト、ビルドが成功している
- [ ] 必要な自動テストと実機確認が完了している
- [ ] README、仕様書、リリースノートが更新されている
- [ ] ADRが必要か確認した
- [ ] 権限、通信先、配布物に意図しない変更がない
- [ ] 秘密情報や個人情報が含まれていない
- [ ] `package.json`を未公開のSemVerへ更新した
- [ ] `pnpm release:prepare`が成功している

`main`へのマージ後は、GitHub Actionsが同じバージョンのReleaseを作成します。既に公開済みのバージョンは再利用できません。
